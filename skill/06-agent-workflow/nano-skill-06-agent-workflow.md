## 一、Agent 的五个组成部分

先画一张简单的全景图：

```text
┌──────────────────────────────────────────────┐
│                   Agent                       │
│                                               │
│  ┌─────────────────────────────────────────┐  │
│  │            模型底座（LLM）               │  │
│  │    理解语言、推理、生成——通用智能        │  │
│  └─────────────────────────────────────────┘  │
│                                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  Rules   │  │  Memory  │  │   MCP    │   │
│  │  性格    │  │  记忆    │  │  手脚    │   │
│  └──────────┘  └──────────┘  └──────────┘   │
│                                               │
│  ┌─────────────────────────────────────────┐  │
│  │              Skills                      │  │
│  │    工作手册——遇到这类任务该怎么做        │  │
│  └─────────────────────────────────────────┘  │
│                                               │
└──────────────────────────────────────────────┘
```

**模型底座**是通用智能——理解语言、逻辑推理、生成内容。它是一个聪明的大脑，但没有任何关于”你的工作”的具体知识。

**Rules**是性格与全局底线——对所有任务永远生效的硬约束。比如”始终用中文回答”“不允许删除生产数据库”。不区分场景，不需要触发条件，写了就永远生效。

**Memory**是个人记忆与上下文——让 Agent 记住”你”的偏好和历史。它知道”上次你让我用 TypeScript 而不是 JavaScript”，保持跨对话的连贯性。

**MCP**是外部执行能力——Agent 的”手脚”，负责真正操作外部系统。GitHub、Slack、数据库、文件系统——没有 MCP，Agent 只能”说”不能”做”。

**Skills**是领域工作手册——针对特定任务类型的标准化流程与最佳实践。它把”聪明但什么都不知道的新员工”变成”知道该怎么干活的熟手”。

五个部分各有分工。混淆了，就会在错误的层面解决问题。

## 二、Skill 和其他组件的边界

这五个组件经常被混淆。一个常见的问题是：这件事该写在 Rules 里还是 Skill 里？该用 MCP 还是 Skill？

### Skill vs Rules：什么时候用哪个？

**Rules 是全局的，Skill 是局部的。**

“回答时始终用中文”——这是 Rules。它对所有任务、所有场景都生效，不需要判断”什么时候触发”。

“遇到公众号链接时，调用 WeSpy 抓取内容并转为 Markdown”——这是 Skill。它只在特定任务出现时才激活，有明确的触发条件、操作步骤和边界。

一个简单的判断方法：**如果这条指令不管 Agent 在做什么任务都应该遵守，放 Rules。如果这条指令只在特定类型的任务中才需要，放 Skill。**

还有一种常见的错误：把应该是 Skill 的东西写进 Rules。比如把”代码审查时要检查安全漏洞、性能问题、代码风格”写进 Rules——这条指令只在代码审查时才需要，写进 Rules 意味着 Agent 在写周报、回邮件的时候也要”遵守”这条规则，徒增上下文负担。它应该是一个独立的代码审查 Skill。

### Skill vs MCP：手脚与工作手册

MCP 是手脚，Skill 是工作手册。

MCP 解决的是“Agent 能不能操作这个系统”——能不能读取 GitHub PR、发送 Slack 消息、查询数据库。它提供连接和执行能力。

Skill 解决的是“Agent 该怎么操作”——PR 审查时该重点看哪些维度、Slack 消息该用什么格式、数据库查询该注意哪些性能陷阱。它把零散的能力组织成一套清晰、可重复的最佳实践流程。

有了 MCP，Agent 具备了执行能力；
而 Skill 则让这些能力用得更专业、更稳定、结果更一致。

一个好的 Skill，能显著提升 Agent 在具体任务上的表现，让同样的工具发挥出更高的价值。

### Skill vs Memory：工作手册和笔记本

Memory 记住的是”上次发生了什么”——你的偏好、历史对话、之前的决策。

Skill 规定的是”这类任务该怎么做”——标准流程、最佳实践、边界条件。

Memory 是个人的、变化的、累积的。Skill 是通用的、稳定的、可共享的。

你不会把”用户喜欢简洁风格”写进 Skill——这是 Memory 的职责。你也不会把”公众号文章抓取的完整流程”存进 Memory——这是 Skill 的职责。

## 三、Skill 在 Agent 架构中的独特价值

理清了边界，再来看 Skill 的独特价值到底在哪。

**在 Agent 的五个组件里，Skill 是唯一一个”边际收益最高、且完全由你掌控”的部分。你投入的时间，几乎可以 1:1 转化为 Agent 在你领域里的专业能力。**

模型底座是厂商提供的，你改不了。Rules 通常比较简单，写几条就够了。Memory 是系统自动管理的。MCP 是工具连接，配置好就完事了。

只有 Skill，是需要你根据自己的业务场景、工作流程、踩过的坑，持续创建和迭代的。而且你创建的每一个 Skill，都在让 Agent 变得更懂你的工作、更能帮上忙。

这就是为什么第一篇说”Skill 是杠杆最高的投入”：

- 换一个更强的模型，效果提升是渐进的
- 多接一个 MCP，Agent 多了一个工具
- 写一个好的 Skill，Agent 可能从”做不到”变成”做得到”——这是从 0 到 1 的突破

而且 Skill 是**可积累、可复用、可分享**的。你写的 Skill 可以给团队用，可以开源给社区，可以在不同的 Agent 工具之间迁移（因为 30+ 工具已经在向同一套目录结构靠拢）。

## 四、一个完整的例子：五个组件怎么协同

说了这么多抽象的概念，用一个完整的例子串起来。

场景：你让 Agent “帮我审查这个 GitHub PR，重点看安全问题，审完在 Slack 通知我”。

Agent 收到这个请求后，五个组件各自扮演什么角色？

**模型底座**：理解你的自然语言请求，判断需要做三件事——拉 PR 代码、审查安全问题、发 Slack 通知。

**Rules**：全局约束生效——比如”始终用中文回复”“不允许直接合并 PR，只能给建议”。

**Memory**：回忆起你之前说过”安全审查结果发到 #security-review 频道”，而不是你的私信。

**MCP**：连接 GitHub（拉取 PR 的文件变更）和 Slack（发送审查结果）。这是”手脚”——没有 MCP，Agent 看不到 PR 代码，也发不了 Slack 消息。

**Skill**：加载”代码安全审查 Skill”。这个 Skill 告诉 Agent：按什么清单检查（注入风险、权限边界、敏感数据处理）、结果按什么格式输出（按严重程度分级）、什么情况下要特别标注（涉及认证逻辑的变更必须标红）。

五个组件缺一不可：

- 没有模型底座 → 不理解你在说什么
- 没有 Rules → 可能直接帮你合并了 PR
- 没有 Memory → 不知道结果该发到哪个频道
- 没有 MCP → 看不到代码，发不了消息
- 没有 Skill → 会审查，但不知道该重点看什么，输出格式每次不一样

这就是全局视角的价值：**理解每个组件的角色，才能在正确的层面解决正确的问题。**

## 五、写给不同角色的人

这个系列的读者背景各不相同。在收官篇，我想给不同角色的人各说一句话。

**如果你是开发者：**

Skill 对你来说是最趁手的工具。你已经习惯了写文档、写配置、写自动化脚本——Skill 不过是把这些经验用 Markdown 格式写给 Agent 看。从你最痛的重复工作开始，写第一个 Skill，用 skill-creator 迭代，然后逐步积累。你的 Skill 库会成为你的技术资产。

**如果你是产品经理或运营：**

别被”技术”两个字吓退。Skill 的本质是”把经验写清楚”——你每天都在做这件事，只不过之前写的是给人看的 SOP，现在写的是给 Agent 看的 SOP。你最擅长的”把复杂流程拆解成清晰步骤”，正是写 Skill 最需要的能力。从一个你反复跟 Agent 重复的需求开始（比如”帮我整理会议纪要”“帮我写周报”），把它固化成 Skill。

**如果你是团队负责人：**

Skill 最大的组织价值不是提升个人效率，而是**让经验可传承**。团队里最资深的人踩过的坑、积累的最佳实践，以前只存在他们的脑子里，靠口口相传。现在可以写成 Skill，让 Agent 帮整个团队执行。一个人的经验，变成了整个团队的能力。

## 六、这个系列讲了什么

回顾一下这个系列的完整脉络：

**第一篇**，我们从一个公众号文章的场景切入，看到了”有 Skill vs 没 Skill”的巨大差距——不是做得好不好的问题，是做得到做不到的问题。Skill 的本质是给 Agent 的工作手册。

**第二篇**，我们拆开 wespy-fetcher 的真实 SKILL.md，逐层解剖了 Skill 的四层结构——头部、概述、操作指南、补充说明。每一层该怎么写才有效，写错了会怎样。

**第三篇**，我们从零写了一个 article-summarizer，经历了三轮迭代：第一版凭直觉，第二版靠知识，第三版靠经验。核心方法论是”不要试图一次写对，快速迭代”。

**第四篇**，我们介绍了 Anthropic 官方的 skill-creator——一个帮你创建、测试和迭代 Skill 的”元技能”，把手工迭代变成有数据支撑的系统化流程。

**第五篇**，我们聊了 Skill 之间怎么组合——接力、分工、编排，以及什么时候该拆、什么时候不该拆。核心原则是单一职责、通用格式、显式边界。

**这篇收官**，我们把 Skill 放回 Agent 的全局架构中，看它和 Rules、Memory、MCP 的关系，理解每个组件的角色和边界。

从”Skill 是什么”到”Skill 在整个 Agent 架构中处于什么位置”，这个系列走完了一条从微观到宏观的完整路径。

## 七、最后一件事

写到这里，这个系列该说的都说了。但最后还想多说一件事。

很多人在等——等更好的模型、等更完善的工具、等别人写好 Skill 自己直接用。

但 Skill 最大的价值恰恰来自**你自己的经验**。

模型会越来越强，工具会越来越好用，但你踩过的坑、你总结的流程、你对业务的理解——这些不会从天上掉下来。它们在你的脑子里，只有你能把它们写成 Skill。

而一旦你写出来了，这份经验就不再只属于你——它属于你的团队、你的 Agent、你构建的整个工作流系统。它会在你不在的时候继续生效，在你想不到的场景里发挥作用。

所以不要等了。打开编辑器，新建一个文件夹，创建一个 SKILL.md。

从你最烦的那件重复工作开始。

**你写下的每一份经验，都会变成 Agent 永久的能力，也会成为你和团队最宝贵的数字资产。**
