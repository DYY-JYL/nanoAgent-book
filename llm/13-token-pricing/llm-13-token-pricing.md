# 从零开始理解大模型（番外篇三）：为什么用 Token 计费？为什么输出比输入贵？

> **「从零开始理解大模型」番外篇** —— 正篇十篇讲原理，番外篇回答那些“懂了原理之后自然会好奇”的问题。

你用大模型 API 的时候，账单上写的不是“调用次数”，而是“token 数”。而且你会发现一个奇怪的现象：

```text
DeepSeek V4 API 定价（单位：元/百万 tokens）：

模型         缓存命中输入    缓存未命中输入    输出
V4-Flash      0.2 元          1 元             2 元
V4-Pro        1 元            12 元            24 元
```

三个自然的问题：

1. 为什么按 token 计费，而不是按次或按字？
2. 输出 token 为什么比输入贵（V4-Pro 贵了整整 2 倍）？
3. “缓存命中”是怎么回事，为什么能便宜 12 倍？

读完正篇的你，已经有了回答这三个问题所需的全部知识。这篇把它们串起来。

-----

## 一、为什么按 Token 计费

### 1.1 因为 Token 是模型真正的“工作量单位”

第二篇讲过，模型不认识字，只认识 token。你输入的每一段文字，都要先切成 token，模型才能处理。

关键在于：**模型的计算量和 token 数成正比，和“字数”或“调用次数”不成正比。**

同样是“一次调用”：

- “你好” → 2-3 个 token → 模型跑一小会儿
- 贴一篇万字长文 → 15,000 个 token → 模型跑很久

如果按次计费，这两种调用收一样的钱，服务商就亏惨了。按字计费呢？第二篇讲过，中文和英文的 token 效率不同——同样的内容中文可能比英文多用 2 倍 token。按字数收费会导致中英文成本对不上。

**按 token 计费最公平——token 数直接反映了模型消耗的算力。** 处理 1000 个 token 就是跑了 1000 步 Transformer，不管这 1000 个 token 对应的是 500 个英文单词还是 600 个汉字。

### 1.2 输入和输出为什么分开计价

因为输入和输出的计算方式完全不同（第七篇）：

|      | 输入（Prefill） | 输出（Decode） |
|------|-----------------|----------------|
| 处理方式 | 一次性并行处理所有 token | 一个一个串行生成 |
| GPU 利用率 | 高（大矩阵乘法，GPU 吃满） | 低（每步只算一个 token） |
| 瓶颈 | 计算量 | 内存带宽 |
| 成本 | 较低 | 较高 |

分开计价让定价更精确——用的资源不同，价格也不同。

## 二、为什么输出 Token 比输入贵

这是最反直觉的一点：输入 token 数量通常比输出多，但**单价**反而是输出更贵。原因全在第七篇讲的 Prefill 和 Decode 的差异。

### 2.1 输入：一口气吃下，GPU 很开心

Prefill 阶段，所有输入 token 一次性送入模型。GPU 擅长的就是这种大规模并行计算——几千个 token 的矩阵乘法，几千个核心同时算，GPU 利用率很高。

打个比方：你给厨师一次性拿来 100 份食材，他用流水线一起处理，效率很高。

### 2.2 输出：一个一个蹦，GPU 很闲

Decode 阶段，每次只生成一个 token。第七篇讲过，每生成一个 token 都要：

1. 把新 token 送入模型
2. 从显存读一遍全部参数（番外篇二讲的内存带宽瓶颈）
3. 用 KV Cache 做 Attention
4. 输出一个 token

**GPU 几千个核心，但每步只在处理一个 token 的计算量**——大部分核心在等着。就像一个大厨房里有 50 个灶台，但每次只能炒一个菜。

### 2.3 算一笔账

以 LLaMA 7B 为例（番外篇二的数据）：

```text
处理 1000 个输入 token（Prefill）：
  1000 个 token 的矩阵乘法一次性做完
  GPU 利用率高，假设 80%
  耗时：约 50ms

生成 1000 个输出 token（Decode）：
  每个 token 都要跑一遍模型（读 14 GB 参数）
  GPU 利用率低，假设 10-20%
  每步约 5-10ms × 1000 步 = 5,000-10,000ms
```

同样 1000 个 token：输入 50ms，输出 5000ms——**输出占用 GPU 的时间是输入的 100 倍**。

GPU 时间就是钱。输出 token 占用 GPU 时间更长，所以更贵。通常输出单价是输入的 2-4 倍。

## 三、缓存命中 Token 是怎么回事

很多 API 的账单上还有一行“缓存命中 token”，单价比普通输入便宜好几倍。这是什么？

### 3.1 什么时候会“命中缓存”

当你多次调用 API，且前面的内容相同时：

```text
第一次调用：
  System: "你是一个翻译助手，请将以下内容翻译成英文。"
  User: "今天天气真好。"

第二次调用：
  System: "你是一个翻译助手，请将以下内容翻译成英文。"   ← 和上次一样！
  User: "这个餐厅的菜很好吃。"
```

两次调用的 System Prompt 完全一样。第二次调用时，这部分的 KV Cache 可以**直接复用**，不需要重新做 Prefill 计算。

### 3.2 为什么能便宜

回忆第七篇：Prefill 阶段要对所有输入 token 跑完整的 Transformer——Q/K/V 投影、Attention 打分、FFN 处理，每一层都要算。这是实打实的计算量。

缓存命中意味着这些 K 和 V **已经算过了**，直接从缓存里拿就行——省掉了全部的 Prefill 计算，只需要付存储成本。

```text
普通输入 token: Embedding → 32 层 Transformer 完整计算 → 得到 K/V
缓存命中 token: 直接从缓存取 K/V → 跳过 Transformer 计算 → 省了 99% 的算力
```

算力省了，价格自然便宜。以 DeepSeek V4-Pro 为例，缓存命中只要 1 元/百万 tokens，普通输入要 12 元——便宜了 12 倍。

### 3.3 哪些场景容易命中缓存

**System Prompt 不变的多轮对话。** 你和 ChatGPT 聊天时，System Prompt（“你是一个有帮助的助手”）每轮都一样。

**批量处理同格式任务。** 比如用同一个 Prompt 模板翻译 100 篇文章，模板部分每次都命中缓存。

**Agent 场景。** Agent 的系统提示和工具定义（那些 JSON Schema，第十篇讲过）在每次 LLM 调用时都一样。Agent 一个任务可能调 5-10 次 LLM，后面几次的系统提示全都命中缓存——这能省不少钱。

### 3.4 前缀缓存的限制

缓存只对**从头开始的连续相同前缀**有效。如果你改了 System Prompt 的一个字，整个缓存就失效了——因为第一个 token 的 K/V 变了，后面所有的 Attention 计算都会变（第四篇讲过，每个位置的向量取决于它前面所有位置）。

```text
可以命中：
  调用 1: [System A] + [User 问题 1]
  调用 2: [System A] + [User 问题 2]     ← System A 命中缓存

不能命中：
  调用 1: [System A] + [User 问题 1]
  调用 2: [System B] + [User 问题 2]     ← System 不同，缓存全部失效
```

另外，缓存不是永久的——**它有有效期**。大多数平台的缓存会在几分钟到几十分钟内过期（具体时长因服务商和负载而异）。如果两次调用之间间隔太久，缓存可能已经被清理了，只能重新计算。所以缓存省钱的前提是**短时间内密集调用**——这恰好是 Agent 场景和批量任务的典型模式。

## 四、一张图串起来

```text
你发送一条 API 请求（以 DeepSeek V4-Pro 为例）

  输入 token（比如 2000 个）
    │
    ├── 前 1500 个和上次一样 → 缓存命中！跳过 Transformer 计算
    │   → 按缓存价 1 元/百万 tokens 计费 → ¥0.0015
    │
    ├── 后 500 个是新内容 → 正常 Prefill 计算
    │   → 按输入价 12 元/百万 tokens 计费 → ¥0.006
    │
    ▼
  输出 token（比如 800 个）
    │
    ├── 每个 token 都要串行生成，每步读一遍模型参数
    │   GPU 利用率低，占用时间长
    │   → 按输出价 24 元/百万 tokens 计费 → ¥0.0192
    │
    ▼
  总费用 = ¥0.0015 + ¥0.006 + ¥0.0192 = ¥0.0267

  其中输出只占 token 总量的 29%，但占了费用的 72%
```

## 五、实用建议

**控制输出长度。** 输出是最贵的部分。如果你只需要一个“是/否”的判断，在 Prompt 里明确说“只回答是或否”——别让模型写一大段解释。

**复用 System Prompt。** 保持 System Prompt 不变，尽量利用前缀缓存。改一个字缓存就全部失效，所以 Prompt 模板确定后别频繁改动。

**不同语言和内容类型的 token 效率不同。** 中文、英文、代码、表格、JSON 在不同模型的 tokenizer 下 token 数可能差异很大。成本敏感时，最好用目标模型的 tokenizer 或 API 返回的 usage 字段实测，而不是凭字数估算。

**Agent 成本要预估。** Agent 每一步都是一次 API 调用，上下文越来越长（第八篇），输入 token 数逐步增长。一个 10 步的 Agent 任务，总 token 消耗可能是普通单轮对话的 20-50 倍。

-----

## 六、结语

Token 计费不是商家故意把定价搞复杂。它精确反映了模型的真实资源消耗：

- **按 token 计费**——因为计算量和 token 数成正比
- **输出比输入贵**——因为 Decode 串行生成，GPU 利用率低，每个 token 占用时间长
- **缓存命中便宜**——因为跳过了 Prefill 的全部计算，只付存储成本

读完正篇十篇再看这些定价规则，你会发现它们不是商业策略，而是工程现实的直接映射——Prefill vs Decode、KV Cache、GPU 利用率，每一项都对应着真实的硬件成本。

> *看懂账单的人，才能真正控制成本。*
