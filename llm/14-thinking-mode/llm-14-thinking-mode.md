# 从零开始理解大模型（番外篇四）：思考模式和非思考模式到底有什么区别？

> **「从零开始理解大模型」番外篇** —— 正篇十篇讲原理，番外篇回答那些“懂了原理之后自然会好奇”的问题。

你用 DeepSeek、ChatGPT 或 Claude 的时候，可能注意到有些模型有两种模式：

- **普通模式**：你问它问题，它直接回答
- **思考模式**（深度思考 / Extended Thinking）：回答之前先有一个“思考中……”的过程，等几秒甚至几十秒，然后给出一个（通常）更好的回答

这个“思考”到底在干什么？模型真的在“想”吗？还是只是故意慢一点？

读完正篇的你，已经有了理解这个问题的全部基础。

-----

## 一、先说结论

|       | 非思考模式 | 思考模式 |
|-------|-----------|---------|
| 输出方式 | 直接生成回答 | 先生成一段“思考过程”，再生成回答 |
| 思考过程可见吗 | 没有思考过程 | 有些模型展示，有些折叠隐藏 |
| 回答速度 | 快 | 慢（思考过程本身也是在生成 token） |
| 适合的任务 | 简单问答、翻译、闲聊 | 数学、逻辑推理、代码、复杂分析 |
| 成本 | 低 | 高（思考过程的 token 也要计费） |
| 底层机制 | 预测下一个 token | 还是预测下一个 token |

一句话版本：**思考模式不是什么新架构。模型做的事情和正篇讲的完全一样——预测下一个 token。区别只是：在给出答案之前，先让模型多生成一些“自言自语”的 token，这些 token 构成了一条推理链，帮助模型在后续 token 中给出更准确的答案。**

-----

## 二、本质：让模型“自言自语”

### 2.1 正篇讲的生成过程

第七篇讲过，模型生成回答的过程是：

```text
输入: "12 × 15 等于多少？"
  │
  ▼ Prefill → Decode → 一个 token 一个 token 蹦出来
  │
输出: "180"
```

模型看到问题后直接输出答案。没有中间过程——或者说，中间过程全在 Transformer 的 32 层计算里，一次前向传播就结束了。

### 2.2 思考模式的生成过程

思考模式下，模型在回答之前先生成一段“思考”：

```text
输入: "12 × 15 等于多少？"
  │
  ▼ Prefill → Decode → 先生成思考过程
  │
思考: "我需要计算 12 × 15。
      可以拆成 12 × 10 + 12 × 5
      12 × 10 = 120
      12 × 5 = 60
      120 + 60 = 180"
  │
  ▼ 继续 Decode → 生成最终回答
  │
输出: "180"
```

**这段“思考”不是什么特殊计算——它就是模型生成的普通 token。** 和生成“从前有座山”没有任何区别，都是一个 token 一个 token 预测出来的。

但它有一个关键效果：**思考过程中产生的 token 会进入上下文窗口，成为后续预测的输入。**

### 2.3 为什么“自言自语”能提高准确率

回忆第四篇讲的 Attention——模型在预测每一个新 token 时，会看上下文中的**所有**已有 token。

没有思考过程时，模型预测 “180” 这个答案时，上下文里只有原始问题 “12 × 15 等于多少？”。模型要在一次前向传播中完成全部推理——32 层 Transformer，走完就得出答案。

有了思考过程之后，模型预测最终答案时，上下文里多了一大段中间步骤——“12 × 10 = 120”、“12 × 5 = 60”、“120 + 60 = 180”。这些中间结果作为 token 已经“写”在了上下文窗口里，模型在做最后的 Attention 时可以直接引用它们。

**思考过程相当于给模型提供了“草稿纸”。** 没有草稿纸，你让一个人心算 12 × 15 可能会出错。给他一张纸让他写下中间步骤，准确率就高多了。模型也是一样——中间步骤写成 token 放在上下文里，后续预测就有了更多的“参考资料”。

## 三、从 Transformer 的角度理解

为什么多几层 Transformer 计算不够，还需要额外生成 token 当草稿纸？

### 3.1 一次前向传播的算力是固定的

第五篇讲过，一次前向传播就是输入经过 N 层（Attention + FFN）。对于一个 32 层的模型，不管问题有多难，它都只做 32 层计算就要给出答案。

简单问题（“1 + 1 等于几”）和复杂问题（“请证明费马大定理”）分配到的算力是一样的——都是 32 层。

**这就像考试只给你 10 秒钟，不管题目难不难都必须交卷。** 简单题 10 秒够了，难题 10 秒根本不够。

### 3.2 多生成 token = 多做几次前向传播

思考模式的巧妙之处在于：**每多生成一个 token，就多做一次完整的前向传播（32 层 Transformer）。**

```text
非思考模式:
  问题 → [32层] → 答案
  总计算量: 1 次前向传播

思考模式（生成了 100 个思考 token）:
  问题 → [32层] → 思考token₁
       → [32层] → 思考token₂    ← 能看到 token₁
       → [32层] → 思考token₃    ← 能看到 token₁₂
       ...
       → [32层] → 思考token₁₀₀  ← 能看到前 99 个思考 token
       → [32层] → 答案           ← 能看到全部 100 个思考 token
  总计算量: 101 次前向传播
```

**思考模式用“生成更多 token”的方式，变相给了模型更多的计算量。** 每一步都是 32 层 Transformer，但做了 101 步而不是 1 步。而且每一步都能看到前面所有步骤的结果——相当于把一个 32 层的模型变成了一个“可以反复查看自己笔记”的 32 × 101 层的推理链。

### 3.3 这就是 Test-time Compute

第九篇 Scaling Law 讲的是**训练时的规模**——参数越多、数据越多、训练算力越大，模型越好。

思考模式代表了另一个维度的 Scaling：**推理时的规模（Test-time Compute）**——同一个模型，推理时给它更多的计算量（让它多生成一些 token 来“想一想”），回答就更好。

```text
Scaling 的两个维度：

训练时 Scaling（第九篇）: 花更多钱训练 → 更好的模型
推理时 Scaling（本篇）:  花更多钱推理 → 同一个模型给出更好的回答
```

这也解释了番外篇三讲的成本问题——思考模式的那些“思考 token”都是要计费的输出 token，而输出 token 是最贵的。

## 四、思考模式是怎么训练出来的

模型不是天生就会“先想再答”的。它需要通过训练学会这种行为。

### 4.1 和正篇第六篇的联系

第六篇讲的三阶段训练：预训练 → SFT → RLHF。思考模式主要在后两个阶段实现：

**SFT 阶段**：用大量“带推理过程的问答对”做微调。训练数据长这样：

```text
用户: 一个水池有两个进水管和一个出水管...（应用题）

助手: <think>
  已知条件：进水管 A 每小时进 3 吨，进水管 B 每小时进 2 吨...
  需要求：多少小时能装满...
  列方程：(3 + 2 - 1) × t = 40
  解方程：4t = 40, t = 10
</think>

水池需要 10 小时装满。
```

模型通过 SFT 学会了：看到这类问题时，先生成 `<think>...</think>` 中的推理步骤，再给出答案。

**RLHF / RL 阶段**：用强化学习优化推理质量。让模型对同一个问题生成多种推理路径，正确答案的路径获得奖励，错误路径获得惩罚。模型逐渐学会“什么样的思考过程更容易导向正确答案”。

### 4.2 思考 token 的格式

不同模型用不同的标记来区分“思考”和“回答”：

```text
DeepSeek V4:   选中“思考模式”后，<think>推理过程</think>最终回答
```

但底层都是一样的——**思考过程就是一段 token，和回答的 token 没有本质区别。** 区别只在于它被标记了“这是思考”，前端可以选择折叠或展示。

## 五、什么时候该用思考模式

思考模式不是万能的。它有明确的适用场景和不适用场景。

### 5.1 适合用思考模式

**多步推理**：数学题、逻辑推理、需要分步分析的问题。中间步骤越多，思考模式的优势越大。

**代码生成和调试**：需要考虑多种边界情况、追踪变量状态、设计架构。

**复杂分析**：需要权衡利弊、考虑多个因素、得出结论的任务。

### 5.2 不适合用思考模式

**简单问答**：“法国首都是哪里？”——不需要推理，直接回答即可。思考模式只会浪费 token 和时间。

**翻译、摘要**：这类任务不需要多步推理，思考过程帮助不大。

**闲聊**：日常对话不需要严密逻辑。

### 5.3 成本差异

```text
同一个问题，不同模式的 token 消耗：

非思考模式:
  输入: 100 tokens
  输出: 200 tokens
  总计: 300 tokens

思考模式:
  输入: 100 tokens
  思考: 800 tokens（思考过程）  ← 这些都是输出 token，按输出价计费！
  输出: 200 tokens
  总计: 1100 tokens
```

思考模式的 token 消耗可能是非思考模式的 3-5 倍，而且额外的部分全是最贵的输出 token（番外篇三讲过）。

## 六、和 Agent 的关系

如果你读过 Agent 系列，会发现思考模式和 Agent 有一个有趣的相似之处：**都是用“多步”来解决“一步搞不定”的问题。**

```text
思考模式: 问题 → 思考token₁ → 思考token₂ → ... → 答案
                 └── 模型内部的多步推理 ──┘

Agent:    问题 → 工具调用₁ → 工具调用₂ → ... → 答案
                 └── 模型 + 外部工具的多步协作 ──┘
```

区别在于：

- **思考模式**的“多步”发生在模型内部——生成更多 token 来推理
- **Agent** 的“多步”涉及外部世界——调用工具、获取信息、再推理

实际中两者经常组合使用：Agent 的每一步决策用思考模式来做更好的判断，思考模式在需要外部信息时触发工具调用。

-----

## 七、结语

思考模式没有任何新的架构或新的算法。它就是正篇讲的那套——预测下一个 token。

唯一的区别是**让模型在回答之前多“说”一些 token**。这些额外的 token 构成了推理链，写在上下文窗口里，让模型在最终回答时有更多的参考信息。

从 Transformer 的角度看，这等于用“多步前向传播”来弥补“单步算力有限”的问题。从 Scaling Law 的角度看，这是在推理阶段做 Scaling——同一个模型，花更多推理算力，得到更好的结果。

> *模型不会“思考”。它只是在预测下一个 token 时，先给自己写了一份草稿。但就是这份草稿，让它从“心算”变成了“笔算”——准确率天差地别。*
