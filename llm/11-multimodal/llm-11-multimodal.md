# 从零开始理解大模型（番外篇）：多模态——大模型是怎么”看懂”图片的

> **「从零开始理解大模型」番外篇** —— 正篇十篇讲的是纯文本大模型。这篇讲多模态：当大模型不只处理文字，还能看图、听声音的时候，架构上到底变了什么？

你问 ChatGPT “这张图片里有什么？”，它能告诉你图里有一只猫坐在沙发上。你发一张手写数学题的照片，它能识别并解题。

但正篇十篇讲的大模型只处理 token——一串数字。**图片不是 token，它怎么进得了 Transformer？**

这篇回答一个核心问题：**多模态大模型和文本大模型的区别到底在哪？**

-----

## 一、先说结论

|              |文本大模型（正篇）  |多模态大模型（本篇）                                |
|--------------|-----------|------------------------------------------|
|输入            |只有文本       |文本 + 图片（+ 音频、视频等）                         |
|Transformer 架构|不变         |**几乎不变**                                  |
|核心改动          |—          |加了一个”翻译器”，把图片翻译成向量序列                      |
|Attention 机制  |不变         |核心机制不变（仍是向量间通过 Attention 建立关系，具体交互方式因模型而异）|
|训练目标          |预测下一个 token|还是预测下一个 token                             |
|输出            |文本         |主要还是文本（部分模型能输出图片）                         |

一句话版本：**多模态大模型 = 文本大模型 + 一个把图片”翻译”成向量的编码器。Transformer 本身几乎没变。**

-----

## 二、核心思路：把一切都变成向量

正篇第三篇讲过，文本进入 Transformer 之前要经过两步：

```
文字 → Token（第二篇）→ 向量（第三篇）→ 进入 Transformer
```

Transformer 内部只操作向量——它不关心这些向量是从文字来的还是从图片来的。只要你能把图片也变成一组向量，就可以和文本向量混在一起，一起送进 Transformer。

**这就是多模态的全部核心思路：给每种模态（文本、图片、音频）各配一个”编码器”，把它们都翻译成同一个向量空间里的向量，然后交给同一个 Transformer 处理。**

```
文本大模型：
  文字 → Tokenizer → Embedding → [向量₁, 向量₂, 向量₃] → Transformer

多模态大模型：
  文字 → Tokenizer → Embedding → [向量₁, 向量₂, 向量₃]  ─┐
                                                          ├→ 拼在一起 → Transformer
  图片 → 图像编码器 → 投影层 → [向量ₐ, 向量ᵦ, 向量ᵧ, ...] ─┘
```

对 Transformer 来说，它看到的就是一个向量序列——前几个来自文字，后面几个来自图片。它用同样的 Attention 机制处理所有向量，不区分来源。

-----

## 三、图片是怎么变成向量的

文本变向量，正篇已经讲透了：文字 → BPE 分词 → Token ID → 查 Embedding 表。

图片的路径不同，但目标一样——变成一组向量。

### 3.1 把图片切成”碎片”

最常用的方法叫 **ViT（Vision Transformer）**，思路非常直接：

```
一张 224×224 的图片
  │
  ▼ 切成 14×14 = 196 个小方块（每个 16×16 像素）
  │
  ▼ 每个小方块通过一个线性层，变成一个向量（比如 768 维）
  │
  ▼ 加上位置编码（告诉模型"这个方块在左上角还是右下角"）
  │
  ▼ 得到 196 个向量
```

这 196 个向量就是图片的”token”。每个向量代表图片的一个局部区域——就像文本的一个 token 代表一个词或子词。

**为什么要切成 patch，而不是把整张图压成一个向量？** 如果把整张 224×224 图直接压成一个向量，会丢失空间细节。切成 196 个 patch 后，每个 patch 保留局部信息，后续 Attention 再让这些 patch 互相”交流”，模型就能同时看到整体和局部——就像文本里词与词之间通过 Attention 建立关系一样。

和文本 token 的对比：

|         |文本           |图片                 |
|---------|-------------|-------------------|
|原始输入     |字符串          |像素矩阵               |
|“分词”方式   |BPE 切成子词     |切成 16×16 的方块（patch）|
|每个”token”|一个子词         |一个图片方块             |
|变成向量     |查 Embedding 表|线性投影（矩阵乘法）         |
|向量维度     |768          |768（对齐到同一维度）       |
|典型数量     |几十到几千        |196～1024 个         |

### 3.2 投影层：对齐到同一个空间

图像编码器（如 ViT）输出的向量和文本 Embedding 的向量通常不在同一个空间里——维度可能不同，语义含义也不同。

需要一个**投影层**（通常就是一两个线性层）把图像向量”翻译”到文本模型的向量空间：

```
图像编码器输出 (1024维) → 投影层 W (1024 → 4096) → 和文本向量同维度 (4096维)
```

投影层的权重是训练时学出来的。训练目标是让投影后的图像向量和对应文本描述的向量尽量接近——“一只猫”的图像向量应该靠近”cat”的文本向量。

## 四、多模态模型的架构：以 LLaVA 为例

LLaVA（Large Language and Vision Assistant）是最经典的多模态架构之一，结构极其清晰：

```
输入："请描述这张图片" + [一张猫的照片]

┌─────────────────────────────────────────────┐
│  文本部分                                     │
│  "请描述这张图片" → Tokenizer → Embedding     │
│  → [向量₁, 向量₂, 向量₃, 向量₄, 向量₅]        │
└──────────────────────┬──────────────────────┘
                       │
                       ▼ 拼接
                       │
┌──────────────────────┴──────────────────────┐
│  图片部分                                     │
│  猫的照片 → ViT 编码器 → 196 个图像向量        │
│  → 投影层 → [向量ₐ, 向量ᵦ, ..., 向量₁₉₆]     │
└──────────────────────┬──────────────────────┘
                       │
                       ▼
         [向量₁, ..., 向量₅, 向量ₐ, ..., 向量₁₉₆]
                       │
                       ▼
              Transformer（和正篇一模一样）
              Attention + FFN × N 层
                       │
                       ▼
              逐 token 生成回答：
              "这张图片中有一只橘色的猫..."
```

注意看：**Transformer 的部分和正篇完全一样。** Attention 公式没变（还是 Q·Kᵀ/√d → softmax → ·V），FFN 没变，残差连接没变，LayerNorm 没变。

唯一的改动是在 Transformer **前面**加了一条图像处理的管线。模型的”大脑”没变，只是”眼睛”是新装的。

-----

## 五、Attention 怎么同时处理文本和图片

你可能会好奇：文本向量和图像向量混在一起，Attention 怎么知道哪些是文字、哪些是图片？

答案是：**它不需要知道。**

第四篇讲过，Attention 做的事情是”让每个向量看看其他所有向量，算出该关注谁”。它不关心向量的来源——只要是一个 d 维的向量，Attention 就能处理。

```
输入序列：[请, 描述, 这张, 图片, patch₁, patch₂, ..., patch₁₉₆]

Attention 矩阵（每个位置和所有位置算相关度）：

              请  描述  这张  图片  p₁  p₂  p₃ ... p₁₉₆
    请       [✓    ·    ·    ·    ·   ·   ·       ·  ]
    描述     [·    ✓    ·    ·    ·   ·   ·       ·  ]
    这张     [·    ·    ✓    ·    ·   ·   ·       ·  ]
    图片     [·    ·    ·    ✓    ·   ·   ·       ·  ]
    patch₁   [·    ·    ·    ·    ✓   ✓   ✓       ✓  ] ← 图像 patch 之间互相看
    patch₂   [·    ·    ·    ·    ✓   ✓   ✓       ✓  ]
    ...
```

生成回答时，模型在预测每个输出 token 时，可以同时关注文本向量（“请描述这张图片”）和图像向量（猫的各个区域）。它学会了：看到”描述”这个词时，要重点关注图像 patch 的内容；看到图片中猫的区域时，要激活和”猫”相关的知识。

**在 LLaVA 这类架构中，跨模态的关注模式不需要额外设计——标准 Attention 自动学会了在文本和图像之间建立关联。** 不过也有一些多模态模型会引入专门的跨注意力模块（Cross-Attention），让图像和文本在特定层进行交互。核心原理相同，实现方式有差异。

## 六、训练：让模型学会”看图说话”

多模态模型的训练分两步，和正篇第六篇的三阶段训练类似：

### 6.1 第一步：预训练对齐

用大量的”图片-文本”配对数据（比如图片和它的描述），训练投影层，让图像向量和文本向量对齐到同一个空间。

这一步通常**冻结 LLM 和图像编码器的权重**，只训练投影层。就像请了一个翻译——LLM 和 ViT 都不用改，只要翻译学会把两种语言对上就行。

### 6.2 第二步：指令微调

用”看图回答问题”的对话数据进行微调。这一步解冻 LLM，让它学会根据图像内容生成详细的回答。

```
训练数据示例：
  用户: [猫的图片] 这是什么动物？
  助手: 这是一只橘色的家猫，它正坐在一张灰色的沙发上...

  用户: [代码截图] 这段代码有什么 bug？
  助手: 第 15 行的循环条件应该是 i < n 而不是 i <= n...
```

**训练目标还是预测下一个 token——和正篇完全一样。** 模型看到文本 token + 图像 token 组成的上下文，预测下一个文本 token。Loss 的计算、梯度下降、反向传播，一模一样。

## 七、多模态带来的新问题

### 7.1 上下文窗口压力更大

正篇第八篇讲过上下文窗口的限制。多模态让这个问题更严重：

```
一段文本:    "请描述这张图片"  →  约 5 个 token
一张图片:    224×224, 16×16 patch → 196 个 token
一张高清图:  448×448, 16×16 patch → 784 个 token
```

一张图片就吃掉了几百个 token 的窗口空间。如果对话中包含多张图片，窗口会被快速占满。

这也是为什么很多多模态模型会对图像做压缩——比如只保留最重要的几十个 patch 向量，而不是全部 196 个。

### 7.2 推理成本更高

更多的 token 意味着更大的 KV Cache（第七篇）和更多的 Attention 计算（n²）。处理一张图片的推理成本远高于处理一句话。

### 7.3 对齐的难度

让图像向量和文本向量在同一个空间里”说同一种语言”，比听起来要难。投影层的质量直接决定了模型”理解”图片的能力。如果投影没做好，模型可能”看到了”图片但”看不懂”。

-----

## 八、不只是图片：音频、视频也是同一个思路

音频、视频的具体编码器实现差别很大，但大方向一致：先各自编码成向量序列，再想办法接到 LLM 上。对当前主流多模态大模型来说，这就是最核心的思路。

下表是常见实现示意：

|模态|编码器（示例）              |“分词”方式           |向量数量    |
|--|---------------------|-----------------|--------|
|文本|Tokenizer + Embedding|BPE 子词           |几十~几千   |
|图片|ViT                  |16×16 patch      |196~1024|
|音频|Whisper 编码器          |短时帧（约 20ms/帧）    |几百~几千   |
|视频|ViT + 时序采样           |每隔几帧取一帧，每帧切 patch|几千~几万   |

**Transformer 是通用的序列处理器。** 它不在乎输入的向量代表的是一个词、一块图片、还是一段声音——只要是向量序列，它就能用 Attention 找出其中的关系。

-----

## 九、总结：变了什么，没变什么

读完正篇十篇再看多模态，你会发现变化出奇地小：

**没变的（正篇的全部核心知识）：**

- Transformer 架构（Attention + FFN × N 层）
- Attention 公式（Q·Kᵀ/√d → softmax → ·V）
- 残差连接、LayerNorm
- 训练目标（预测下一个 token）
- 推理过程（Prefill + Decode + KV Cache）
- Scaling Law（模型越大，多模态能力也越强）

**变了的（只有输入端）：**

- 多了一个图像编码器（ViT）
- 多了一个投影层（把图像向量对齐到文本空间）
- 输入序列从”纯文本 token”变成”文本 token + 图像 token”

**就这些。** 正篇十篇讲的所有东西——从 Token 到 Scaling Law——在多模态模型里全都适用。多模态不是另一套架构，而是在同一套架构的入口处多接了几根管道。

> *大模型的核心能力——理解序列中的关系——不依赖于序列里装的是什么。文字、图片、声音，进了 Transformer 都是向量。这种通用性，才是 Transformer 真正强大的地方。*
