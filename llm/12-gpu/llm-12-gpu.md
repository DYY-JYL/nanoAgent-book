# 从零开始理解大模型（番外篇二）：为什么大模型离不开 GPU

> **「从零开始理解大模型」番外篇** —— 正篇十篇讲的是大模型的原理。这篇回答一个很多人好奇的问题：为什么训练和推理都要用 GPU？CPU 不行吗？

你一定听过这些说法：

- “训练 GPT-4 用了几万张 A100”
- “一张 H100 卖 20 多万”
- “英伟达市值涨了好几倍”

但你有没有想过一个基本问题：**为什么是 GPU？CPU 不是也能做计算吗？**

这篇把这件事讲清楚。读完正篇的你，已经知道大模型内部做的是什么运算了——现在我们看看这些运算为什么偏偏适合 GPU。

-----

## 一、先说结论

|      | CPU | GPU |
|------|-----|-----|
| 核心数量 | 几个到几十个 | 几千到几万个 |
| 单核能力 | 很强，能做复杂逻辑 | 很弱，只能做简单运算 |
| 擅长的事 | 复杂的、有先后顺序的任务 | 简单的、能同时做的大量任务 |
| 类比 | 一个数学教授 | 一万个小学生 |
| 大模型计算 | 能跑，但极慢 | 快几百到几千倍 |

一句话版本：**大模型的核心运算是矩阵乘法，矩阵乘法可以拆成大量独立的乘加运算同时做。GPU 有几千个核心，正好能同时做这些运算。CPU 核心太少，只能一个一个算。**

-----

## 二、大模型到底在算什么

回顾正篇，大模型从头到尾做的运算无非就这几种：

| 运算 | 出现在哪 | 本质 |
|------|----------|------|
| Embedding 查表 | 第三篇 | 按索引取一行向量 |
| Q/K/V 投影 | 第四篇 | 矩阵 × 向量 |
| Attention 打分 | 第四篇 | 矩阵 × 矩阵 |
| Softmax | 第四篇 | 逐元素 exp + 求和 + 除法 |
| FFN | 第五篇 | 矩阵 × 向量 × 2 |
| LayerNorm | 第五篇 | 求均值 + 求方差 + 逐元素运算 |
| 残差连接 | 第五篇 | 向量 + 向量 |

看出规律了吗？**绝大部分运算都是矩阵乘法。**

第五篇讲过，一层 Transformer 就是：Attention（矩阵乘法 × 4）+ FFN（矩阵乘法 × 2）+ 一些逐元素运算。32 层 Transformer 就是把这些矩阵乘法重复 32 遍。

所以问题变成了：**为什么矩阵乘法适合 GPU？**

-----

## 三、矩阵乘法为什么适合 GPU

### 3.1 矩阵乘法是什么

两个矩阵相乘，结果矩阵的每一个元素都是一行和一列的“点积”：

```text
结果[i][j] = A 的第 i 行 · B 的第 j 列
           = A[i][0]×B[0][j] + A[i][1]×B[1][j] + ... + A[i][d]×B[d][j]
```

如果 A 是 4096×4096，B 也是 4096×4096，结果矩阵有 4096×4096 = 约 1680 万个元素。每个元素需要 4096 次乘法和 4096 次加法。

总运算量：1680 万 × 4096 × 2 ≈ **1370 亿次运算**。

一层 Transformer 里有好几次这样的矩阵乘法。32 层就是好几十次。每生成一个 token 都要跑一遍。

### 3.2 关键特性：每个元素的计算是独立的

`结果[0][0]` 的计算不需要等 `结果[0][1]` 算完。1680 万个元素可以**同时算**，互不依赖。

这就是矩阵乘法的核心特性：**大量独立的、相同的运算。**

### 3.3 CPU vs GPU 怎么处理这件事

**CPU（数学教授）**：

```text
CPU 有 16 个核心
1680 万个元素 ÷ 16 个核心 = 每个核心算 105 万个元素
每个元素需要 4096 次乘加
→ 串行地一个一个算，需要很长时间
```

CPU 的每个核心很强——能做复杂的分支判断、循环嵌套、内存管理。但矩阵乘法不需要这些复杂能力，它只需要“乘完再加”。CPU 的强大在这里是浪费的。

**GPU（一万个小学生）**：

```text
GPU 有 16384 个核心（以 A100 为例）
1680 万个元素 ÷ 16384 个核心 = 每个核心只算约 1000 个元素
→ 大家同时算，一瞬间就完事了
```

GPU 的每个核心很弱——只能做简单的乘法和加法，复杂逻辑不行。但矩阵乘法恰好只需要乘法和加法。数量碾压弥补了单核能力的不足。

### 3.4 用数字感受差距

以 LLaMA 7B 推理为例（生成一个 token）：

```text
总计算量:    约 140 亿次浮点运算（约 14 GFLOP）

CPU（Intel i9, 算力约 1 TFLOPS）:
  14 GFLOP ÷ 1000 GFLOP/s ≈ 14ms
  但实际受限于内存带宽，可能需要 200-500ms

GPU（A100, 算力约 312 TFLOPS）:
  14 GFLOP ÷ 312,000 GFLOP/s ≈ 0.045ms
  实际考虑内存访问，约 5-10ms
```

差了几十到上百倍。而且这只是一个 token——生成一段 500 token 的回答，差距就是“1 秒 vs 几分钟”。实际体验是：GPU 上回答几乎是瞬时蹦出来的，CPU 上生成一句话可能要等十几秒甚至几十秒。

-----

## 四、不只是算力：内存带宽也是关键

计算快还不够。数据得能快速喂到计算核心里。

### 4.1 什么是内存带宽

矩阵乘法需要从显存里把矩阵数据读出来，送到计算核心，算完再写回去。这个“搬运数据”的速度叫**内存带宽**。

```text
CPU 内存带宽（DDR5）:   约 80 GB/s
GPU 显存带宽（A100）:   约 2,000 GB/s    ← 25 倍！
GPU 显存带宽（H100）:   约 3,350 GB/s    ← 42 倍！
```

LLaMA 7B 的参数有 14 GB（float16）。生成每个 token 时，这 14 GB 参数都要从显存读一遍（第七篇讲过）。

```text
读取 14 GB 参数:
  CPU 内存:  14 GB ÷ 80 GB/s = 175ms
  A100 显存: 14 GB ÷ 2000 GB/s = 7ms
```

所以即使计算能力一样快，光“把数据搬到计算核心”这一步，GPU 就比 CPU 快 25 倍。

### 4.2 Decode 阶段瓶颈在带宽，不在算力

第七篇讲过，Decode 阶段每步只处理一个新 token——计算量不大，但需要把整个模型的参数读一遍。这时候瓶颈不是“算得多快”，而是“数据搬得多快”。

这就是为什么 H100 比 A100 贵那么多——不只是计算核心更多，更重要的是显存带宽从 2TB/s 提到了 3.35TB/s。

-----

## 五、GPU 的显存：另一个硬约束

第七篇算过 KV Cache 的显存，第六篇算过训练的显存。GPU 的显存大小直接决定了你能跑多大的模型。

```text
常见 GPU 显存:
  消费级（RTX 4090）:      24 GB  → 能跑 7B 模型（int4 量化）
  专业级（A100 80GB）:     80 GB  → 能跑 70B 模型（int4 量化）
  最新旗舰（H100 SXM）:    80 GB  → 能跑 70B（更快）
  多卡并联（8×H100）:     640 GB  → 能跑 400B+ 模型
```

**模型参数 + KV Cache + 运行时内存，全部要装进显存里。** 装不下就跑不了——这是比算力更硬的约束。

第七篇讲过量化（float16 → int4）能把显存需求压缩 4 倍。这就是为什么量化技术这么重要——不是因为算得更快（虽然也更快），而是因为**装得下了**。

-----

## 六、训练 vs 推理：GPU 的需求完全不同

|      | 训练 | 推理 |
|------|------|------|
| 核心需求 | 算力（大量矩阵乘法） | 带宽（搬运模型参数） |
| 显存需求 | 极大（参数+梯度+优化器+激活值） | 较大（参数+KV Cache） |
| GPU 数量 | 几千到几万张 | 1 到几张 |
| 瓶颈 | 计算量 | 内存带宽 |
| 耗时 | 几周到几个月 | 每 token 几毫秒 |

第六篇讲过，训练一个 7B 模型需要存参数（14GB）+ 梯度（14GB）+ 优化器状态（56GB）+ 激活值（几十GB），总共远超 100GB——一张卡根本装不下，需要分布式训练。

推理相对轻量，但也不轻松。第七篇讲过，KV Cache 会随上下文长度线性增长，128K 上下文的 KV Cache 可能达到 64GB。

-----

## 七、用正篇的知识串一遍

现在你可以完整理解“为什么大模型需要 GPU”了：

**第三篇（Embedding）**：查表取向量——这一步很快，不是瓶颈。

**第四篇（Attention）**：Q·Kᵀ 是矩阵乘法，n 个 token 就是 n×d 和 d×n 的矩阵相乘。n 越大（第八篇的上下文窗口），计算量 n² 增长越猛。GPU 的几千个核心同时算这些点积。

**第五篇（FFN）**：W₁·x 和 W₂·x 是两次大矩阵乘法。FFN 占了模型 45% 的参数——**近一半的计算量在 FFN 里。** 这些矩阵乘法的每个元素都可以独立算，完美适配 GPU。

**第七篇（推理）**：Prefill 阶段计算密集——大矩阵乘法，GPU 算力碾压。Decode 阶段带宽密集——每步要读一遍模型参数，GPU 显存带宽碾压。

**第九篇（Scaling Law）**：Scaling Law 说模型越大越好。模型越大 = 矩阵越大 = 需要更多 GPU。这就是为什么 AI 公司在疯狂采购 GPU——Scaling Law 告诉他们，买更多 GPU = 模型更强 = 更强的产品。

-----

## 八、结语

大模型离不开 GPU，不是因为 GPU 有什么“AI 魔法”，而是因为大模型的核心运算——矩阵乘法——恰好是 GPU 最擅长的事情。

大量独立的、简单的、相同的运算，同时做。GPU 几千个核心并行处理，比 CPU 十几个核心一个一个算快几十到几百倍。再加上显存带宽的巨大优势，GPU 成了大模型时代的“水电煤”。

> *CPU 是一个什么都会的全能选手。GPU 是一万个只会做乘法的工人。大模型需要的恰好是后者——不需要聪明，只需要人多、动作快。*
