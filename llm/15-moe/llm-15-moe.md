# 从零开始理解大模型（番外篇五）：DeepSeek V4 Pro 总参数 1.6T，激活参数 49B——这到底是什么意思？

> **「从零开始理解大模型」番外篇** —— 正篇十篇讲原理，番外篇回答那些“懂了原理之后自然会好奇”的问题。

DeepSeek V4 Pro 发布时，参数表上有两个数字：

```text
总参数:   1.6T（1.6 万亿）
激活参数: 49B（490 亿）
```

总参数 1.6 万亿，但激活参数只有 490 亿——差了 30 多倍。这两个数字分别是什么意思？为什么不一样？

答案藏在一种叫 **MoE（Mixture of Experts，混合专家）** 的架构里。读完正篇第五篇的你，理解它只需要改一个地方。

-----

## 一、先说结论

|      | 标准模型（如 LLaMA 7B） | MoE 模型（如 DeepSeek V4 Pro） |
|------|------------------|-------------------------|
| FFN 数量 | 每层 1 个，所有 token 共用 | 每层几十个“专家”，每个 token 只用其中几个 |
| 总参数 | 7B | 1.6T（所有专家加起来） |
| 激活参数 | 7B（= 总参数） | 49B（每个 token 实际用到的） |
| 推理速度 | 和参数量成正比 | 接近 49B 模型的速度，而非 1.6T |
| 知识容量 | 受限于 7B | 1.6T 的存储空间 |

一句话版本：**MoE 在每层放了几十个 FFN（“专家”），每个 token 只挑其中几个来用。总参数很大（知识容量大），但每次实际计算的参数很少（推理快、成本低）。**

-----

## 二、从第五篇出发：改动只有一个地方

第五篇讲过，一层 Transformer 的结构是：

```text
输入 → LayerNorm → Attention → 残差 → LayerNorm → FFN → 残差 → 输出
```

**MoE 只改了 FFN 这一个位置。** Attention、LayerNorm、残差连接全都不变。

标准模型：

```text
输入 → ... → Attention → ... → FFN → ... → 输出
                                 ↑
                           只有 1 个 FFN
                        所有 token 都走它
```

MoE 模型：

```text
输入 → ... → Attention → ... → Router → 挑 2 个专家
                                          │
                                   Expert 1 (FFN)
                                   Expert 2 (FFN)  ← 被选中
                                   Expert 3 (FFN)
                                   Expert 4 (FFN)  ← 被选中
                                   ...
                                   Expert 64 (FFN)
                                          │
                                    加权合并 → 输出
```

每个 Expert 就是一个独立的 FFN（第五篇讲的 W₁ 扩展 + GELU + W₂ 压缩），结构完全一样，只是权重不同。

**Router（路由器）** 是一个很小的网络（通常就是一个线性层 + Softmax），根据当前 token 的向量决定“这个 token 该交给哪几个专家”。

-----

## 三、总参数 vs 激活参数

理解了 MoE 的结构，这两个数字就很清楚了。

### 3.1 总参数 = 所有专家的参数加起来

```text
标准模型（每层 1 个 FFN）：
  Attention 参数 + 1 个 FFN 参数 = 每层参数
  每层参数 × 层数 + Embedding = 总参数

MoE 模型（每层 64 个专家）：
  Attention 参数 + 64 个 FFN 参数 + Router 参数 = 每层参数
  每层参数 × 层数 + Embedding = 总参数
```

64 个专家的参数全部算进总参数里——所以总参数很大（1.6T）。这些参数都存在显存里，都占空间。

### 3.2 激活参数 = 每个 token 实际用到的

每个 token 只被路由到 2 个专家（64 选 2），其他 62 个专家的参数完全不参与计算。

```text
每个 token 实际计算:
  Attention（全部参与）+ 2 个 FFN（64 个里只选 2 个）+ Router
  ≈ Attention + 2/64 的 FFN 参数
  = 49B
```

**激活参数决定了每个 token 的计算量——推理速度和成本和激活参数成正比，而不是和总参数成正比。**

### 3.3 一个直觉

想象一个大型图书馆（总参数 1.6T）——里面有几十万本书。你去查一个问题时，不需要翻遍所有书，只需要找到最相关的两三本（激活参数 49B）翻一翻就够了。

图书馆的藏书量决定了它“知道多少”。你每次实际翻的书决定了“查一个问题要多久”。

## 四、Router 怎么决定“该找哪个专家”

Router 的工作方式很简单：

```python
# 简化版 Router
def route(token_vector, expert_count=64, top_k=2):
    # 1. 用一个线性层给每个专家打分
    scores = linear_layer(token_vector)    # → 64 个分数

    # 2. 选分数最高的 top_k 个专家
    top_experts = top_k_select(scores, k=2)

    # 3. 对选中的专家做 Softmax，得到权重
    weights = softmax(top_experts.scores)

    # 4. 用权重加权合并选中专家的输出
    output = weights[0] * expert_1(token_vector) + weights[1] * expert_2(token_vector)
    return output
```

这和第四篇 Attention 的打分机制很像——先打分，再选最相关的，最后加权合并。只不过 Attention 是在 token 之间选，Router 是在专家之间选。

### 4.1 不同 token 可能走不同专家

```text
"法国的首都是巴黎" 这句话：

  "法国" → Router 打分 → 选中 Expert 7（地理知识）+ Expert 23（实体识别）
  "首都" → Router 打分 → 选中 Expert 7（地理知识）+ Expert 41（关系推理）
  "巴黎" → Router 打分 → 选中 Expert 7（地理知识）+ Expert 15（城市信息）
```

不同的 token 根据自身内容被路由到不同的专家组合。这意味着模型可以**针对不同类型的内容调用不同的“知识模块”**。

当然，Expert 7 不是真的被命名为“地理知识”——这些专业分工是训练过程中自动形成的，就像第四篇里 Attention 的不同头自动学会关注不同方面一样。

## 五、MoE 的优势和代价

### 5.1 优势：用小算力获得大知识

这是 MoE 最核心的价值，也是 DeepSeek 能把 API 价格打下来的原因之一：

```text
知识容量:  和总参数成正比 → 1.6T 级别 → 知道的东西非常多
推理成本:  和激活参数成正比 → 49B 级别 → 算起来没那么慢
```

用第九篇 Scaling Law 的视角看：**总参数大 → Loss 低（模型能力强），激活参数小 → 推理快、成本低。** MoE 让你在 Scaling Law 曲线上获得大模型的能力，同时只付小模型的推理账单。

### 5.2 代价：显存和通信

**显存问题**：虽然每个 token 只用 2 个专家，但 **所有 64 个专家的参数都要加载到显存里**——因为不同 token 可能路由到不同专家，你不知道下一个 token 会用哪个。所以 MoE 模型的显存需求和**总参数**成正比，不是和激活参数成正比。

```text
DeepSeek V4 Pro 的显存需求:
  总参数 1.6T × 2 字节(float16) ≈ 3.2 TB 显存
  → 需要几十张甚至上百张 GPU 来装
```

**通信问题**：在多 GPU 部署时，不同专家可能分布在不同的 GPU 上。Router 选完专家后，token 的数据要发送到对应专家所在的 GPU，算完再传回来。这个跨 GPU 通信的延迟是 MoE 工程上最大的挑战之一。

### 5.3 负载均衡

如果所有 token 都路由到同一个专家，其他专家就浪费了。训练时会加一个**负载均衡损失（Load Balancing Loss）**，鼓励 Router 把 token 均匀分配到不同专家。

就像餐厅引导客人均匀入座，而不是全挤在一个区域。

## 六、MoE 和正篇知识的关系

| 正篇知识 | 在 MoE 中怎么体现 |
|---------|------------------|
| FFN 是“知识库”（第五篇） | MoE 把一个大知识库拆成几十个小的专业知识库，按需调用 |
| W₁/W₂ 所有 token 共用（第五篇） | MoE 中每个专家有自己的 W₁/W₂，不同 token 可能用不同专家的 W₁/W₂ |
| Attention 不变（第四篇） | MoE 只改了 FFN，Attention 部分和标准模型完全一样 |
| 训练目标不变（第六篇） | 还是预测下一个 token，只是多了一个负载均衡的辅助损失 |
| 推理成本和 token 成正比（第七篇） | MoE 的推理成本和**激活参数**× token 数成正比 |
| Scaling Law（第九篇） | MoE 让总参数和激活参数解耦——可以独立 scale 知识容量和推理成本 |

-----

## 七、结语

MoE 的核心思想极其简洁：**把每层的 FFN 从 1 个变成几十个，每次只激活其中几个。**

就这一个改动，让总参数和激活参数实现了“解耦”——你可以用海量参数存储知识（1.6T），却只付出小模型的推理成本（49B）。这也是为什么 MoE 正在成为越来越主流的大模型架构之一。

> *标准模型像一个全科医生——什么都自己看。MoE 模型像一家大医院——有几十个科室的专家，挂号台（Router）根据你的症状把你分到最合适的科室。医院“总人数”很多，但你每次看病只需要见两三个医生。*
