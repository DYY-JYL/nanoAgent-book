04. SubAgent 子智能体

当单个 Agent 需要同时承担架构、后端、前端工作时，应将部分任务委派给更聚焦的子代理。

SubAgentDelegationRole Prompt
从场景开始 直接看代码

Scenario

什么时候会用到

一个项目同时要做后端、前端、还要写文档，主 Agent 一边在改 Python 一边在改 HTML，上下文越拉越长，最后什么都做不好。把后端的活交给一个专门写后端的子代理，主 Agent 才能集中做协调。

Code First

先看关键代码

聚焦决定行为的核心实现，再结合终端结果对照，能更直观地理解 Agent 的工作方式。

将 subagent 封装为工具

独立 sub_messages 与禁止递归，是这段实现最值得关注的两点。

 107 def subagent(role, task):
 108     """启动一个独立的 Agent 循环，拥有专属角色和独立上下文"""
 109     print(f"\n{'=' * 50}")
 110     print(f"[SubAgent:{role}] 开始: {task}")
 111     print(f"{'=' * 50}")
 112
 113     sub_messages = [
 114         {
 115             "role": "system",
 116             "content": f"You are a {role}. Be concise and focused. Only do what is asked.",
 117         },
 118         {"role": "user", "content": task},
 119     ]
 120     # SubAgent 不能再派 subagent（防无限递归），只用基础工具
 121     sub_tools = [t for t in tools if t["function"]["name"] != "subagent"]
 122
 123     for _ in range(10):
 124         response = client.chat.completions.create(
 125             model=MODEL, messages=sub_messages, tools=sub_tools
 126         )
 127         message = response.choices[0].message
 128         sub_messages.append(message)
 129
 130         if not message.tool_calls:
 131             print(f"[SubAgent:{role}] 完成\n")
 132             return message.content
 133
 134         for tc in message.tool_calls:
 135             fn = tc.function.name
 136             args = json.loads(tc.function.arguments)
 137             print(
 138                 f"  [SubAgent:{role}] {fn}({json.dumps(args, ensure_ascii=False)[:80]})"
 139             )
 140             result = available_functions[fn](**args)
 141             sub_messages.append(
 142                 {"role": "tool", "tool_call_id": tc.id, "content": result}
 143             )
 144
 145     return "SubAgent: max iterations reached"

Live Demo

再看它怎么跑

委派并不神秘，本质就是把另一个 Agent 也封装成工具。

演示命令

python agent/04-subagent/agent-subagent.py "不要直接完成任务。请调用 subagent 工具两次，两个子代理都不要读写文件：1）role=Python API 设计师，task=为 TODO 应用设计 3 个后端接口，只返回接口清单；2）role=前端交互设计师，task=为 TODO 应用设计 3 个界面交互，只返回交互清单。最后主 Agent 用纯文本 4 行汇总，不要表格：后端交付、前端交付、为什么适合委派、主 Agent 没做什么。"
  • 终端会连续出现两次 [Tool] subagent(...)，说明主 Agent 真的把任务委派出去了。
  • 两段日志会分别显示 [SubAgent:Python API 设计师] 与 [SubAgent:前端交互设计师]，角色边界一眼可见。
  • 最终回答只做 4 行汇总，不展开子代理内部历史，突出“主 Agent 负责协调，SubAgent 负责专门任务”。

Try It

自己试一轮

使用 API 设计与前端交互两个轻量角色演示，避免文件生成噪音，让委派过程更清楚。

  1. 将一个写 README 的任务改由文档子代理完成。
  2. 为子代理补充更具体的角色描述，观察输出稳定性的变化。
  3. 让主代理仅保留协调职责，避免同时承担所有实现细节。

Takeaway

这一讲会看懂什么

  • 理解 SubAgent 的关键在于独立上下文，而非额外启动一个模型实例。
  • 能解释为何应禁止子代理继续派生子代理。
  • 能将一个复杂任务拆解为主代理与子代理两类角色。

Key Points

三个核心要点

  • 委派的收益来自上下文收敛，并非单纯的并行化。
  • 角色 prompt 越具体，子代理越容易形成稳定的任务视角。
  • 返回摘要而非全量历史，是后续控制上下文成本的关键习惯。

Pitfalls

容易误解

  • 任务边界不清时，主代理与子代理会出现重复劳动。
  • 允许无限递归委派，成本与复杂度都会失控。
  • 子代理角色过于宽泛时，它只是换了名字的主代理。
