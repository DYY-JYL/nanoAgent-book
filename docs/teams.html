05. 多智能体团队协作

SubAgent 仍属一次性角色；本讲将其升级为具备身份、通信与复盘能力的持久团队。

TeamInboxLifecycle
从场景开始 直接看代码

Scenario

什么时候会用到

希望团队里有一位“开发”和一位“审查”长期存在：每个新任务都跑同样的流程，开发完成后自动转给审查，审查的意见再反馈回开发。SubAgent 那种一次性角色已经不够用了。

持久化的 Agent 对象

这段代码解释了团队成员为何能记得队友此前说过的内容。

看 GitHub 行号 
 145 # ==================== 核心 1: 持久智能体（Agent 类） ====================
 146 #
 147 # 对比 SubAgent（一个函数调用就消亡），Agent 是一个有状态的对象:
 148 #   - 有名字（name）和角色（role）—— 身份
 149 #   - 有 messages 列表 —— 记忆，跨多次 chat() 调用持久保持
 150 #   - 有 inbox —— 通信通道，接收其他 Agent 发来的消息
 151
 152
 153 class Agent:
 154     def __init__(self, name, role):
 155         self.name = name
 156         self.role = role
 157         self.inbox = []  # 通信通道：其他 Agent 发来的消息
 158         self.chat_count = 0
 159         self.messages = [  # 持久记忆：跨多次 chat() 保持
 160             {
 161                 "role": "system",
 162                 "content": f"You are {name}, a {role}. Be concise and focused.",
 163             }
 164         ]
 165         print(f"  [创建] {name} ({role})")
 166
 167     def receive(self, sender, message):
 168         """核心 3: 通信通道 —— 接收来自其他 Agent 的消息"""
 169         self.inbox.append({"from": sender, "content": message})
 170
 171     def chat(self, task):
 172         """
 173         核心 1: 持久记忆 —— 每次 chat() 的对话都累积在 self.messages 中
 174         第二次 chat() 时，Agent 还记得第一次做了什么
 175         """
 176         self.chat_count += 1
 177         print(
 178             f"  [记忆] {self.name} 第 {self.chat_count} 次 chat，"
 179             f"已有 {len(self.messages)} 条 messages，inbox {len(self.inbox)} 条"
 180         )
 181         # 如果 inbox 有新消息，先注入
 182         if self.inbox:
 183             print(f"  [收件箱] {self.name} 读取 {len(self.inbox)} 条团队消息")
 184             mail = "\n".join(f"[来自 {m['from']}]: {m['content']}" for m in self.inbox)
 185             self.messages.append(
 186                 {"role": "user", "content": f"你收到了团队成员的消息:\n{mail}"}
 187             )
 188             # 让 Agent 先消化这些消息
 189             resp = client.chat.completions.create(model=MODEL, messages=self.messages)
 190             self.messages.append(resp.choices[0].message)
 191             self.inbox.clear()
 192
 193         # 执行本次任务
 194         self.messages.append({"role": "user", "content": task})
 195
 196         for _ in range(10):
 197             response = client.chat.completions.create(
 198                 model=MODEL, messages=self.messages, tools=tools
 199             )
 200             message = response.choices[0].message
 201             self.messages.append(message)
 202
 203             if not message.tool_calls:
 204                 print(f"  [{self.name}] → {message.content[:100]}...")
 205                 print(
 206                     f"  [记忆] {self.name} 本轮结束，messages {len(self.messages)} 条"
 207                 )
 208                 return message.content
 209
 210             for tc in message.tool_calls:
 211                 fn = tc.function.name
 212                 args = json.loads(tc.function.arguments)
 213                 print(
 214                     f"  [{self.name}] {fn}({json.dumps(args, ensure_ascii=False)[:60]})"
 215                 )
 216                 result = available_functions.get(fn, lambda **_: "Tool not found")(
 217                     **args
 218                 )
 219                 self.messages.append(
 220                     {"role": "tool", "tool_call_id": tc.id, "content": result}
 221                 )
 222
 223         return "Max iterations reached"

Team 管理生命周期与通信

招募、广播、解散——这里是多智能体协作的最小骨架。

看 GitHub 行号 
 226 # ==================== 核心 2: 身份与生命周期管理（Team 类） ====================
 227 #
 228 # Team 管理 Agent 的完整生命周期:
 229 #   创建（hire）→ 存活（多次 chat + 互相通信）→ 解散（disband）
 230
 231
 232 class Team:
 233     def __init__(self):
 234         self.agents = {}  # name → Agent
 235
 236     def hire(self, name, role):
 237         """招募：创建一个持久 Agent"""
 238         agent = Agent(name, role)
 239         self.agents[name] = agent
 240         return agent
 241
 242     def send(self, from_name, to_name, message):
 243         """核心 3: Agent 之间的通信通道"""
 244         if to_name not in self.agents:
 245             return f"Error: {to_name} not found"
 246         self.agents[to_name].receive(from_name, message)
 247         print(f"  [消息] {from_name} → {to_name}: {message[:60]}...")
 248
 249     def broadcast(self, from_name, message):
 250         """广播：给团队所有其他人发消息"""
 251         for name, agent in self.agents.items():
 252             if name != from_name:
 253                 agent.receive(from_name, message)
 254         print(f"  [广播] {from_name} → 全体: {message[:60]}...")
 255
 256     def disband(self):
 257         """解散：所有 Agent 生命周期结束"""
 258         names = list(self.agents.keys())
 259         self.agents.clear()
 260         print(f"  [解散] 团队已解散 ({', '.join(names)})")

Try It

自己试一轮

用发布前评审流水线演示：开发交付、安全审查、发布 reviewer 复盘，团队价值会更明显。

  1. 将 plan_team() 固定为两开发加一审查的三人团队。
  2. 在执行过程中手动插入一次 send()，对照点对点消息与广播的差异。
  3. 让 reviewer 再次执行 chat()，体会持久记忆如何支撑二次审查。

Takeaway

这一讲会看懂什么

  • 理解 Team 相对 SubAgent 增加的是持久身份与通信。
  • 能用 inbox 这一极简模型解释 Agent 间的消息传递。
  • 理解为何 reviewer 是体现团队价值最直观的角色。

Key Points

三个核心要点

  • 多智能体的关键不在人数，而在角色与生命周期。
  • Agent 的 inbox 模型十分简洁，却已能支撑大量协作场景。
  • 团队规模并非越大越好，信息流的清晰度更具价值。

Pitfalls

容易误解

  • 角色过于宽泛时，团队只是多个普通助手轮流发言。
  • 消息过多时会再次触发上下文压力。
  • 缺少 reviewer 时，难以体现协作带来的质量提升。
