Scenario

什么时候会用到

让 Agent 清理临时文件时，如果它一句 rm -rf 把项目目录也带走，或者读到一个超长文件直接把上下文撑爆，问题就会很具体。能力越强，越需要在它和系统之间留出一道护栏。

再看它怎么跑

建立明确的工程直觉：能力越强，越需要在人机边界处加上护栏。

演示命令

python agent/07-safety/agent-safe.py "列出当前目录的文件"
  • 普通命令会先经过确认再执行。
  • 危险命令在黑名单阶段会被直接拦截。
  • 超长输出会被截断，与上一讲的上下文控制问题相互呼应。

Try It

自己试一轮

  1. 为 DANGEROUS_PATTERNS 增补一条高危命令规则。
  2. 把 read_file 调整为仅可读取项目目录，构建最小允许列表。
  3. 尝试读取超长文件，观察截断提示如何返回给模型。

Takeaway

这一讲会看懂什么

  • 能说清三道防线各自解决的问题。
  • 理解为何不宜将“是否安全”的判断完全交给模型。
  • 理解安全与上下文控制是同一件工程责任的两面。

Key Points

三个核心要点

  • 黑名单拦截已知高危动作，人工确认守住最后边界。
  • 输出截断既是安全问题，也是稳定性问题。
  • 更可控的 Agent 应允许用户随时拒绝某一步。

Pitfalls

容易误解

  • 仅依赖黑名单并不充分，绕过方式始终存在。
  • 所有动作都需确认会显著降低体验，应采用分级策略。
  • --auto 仅适合在完全信任的隔离环境中使用。
