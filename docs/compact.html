06. 上下文压缩

本讲跳过理论铺垫，专注于一个核心问题：长任务中 Agent 为何会被自身历史拖垮，以及压缩如何缓解。

Scenario

什么时候会用到

交给 Agent 一个跨几十轮的长任务，会发现它越走越慢、回答越来越含糊——背后是 messages 一直在膨胀，前面所有工具调用结果都被一次次带着走，最终撑爆 context window。

Code First

先看关键代码

聚焦决定行为的核心实现，再结合终端结果对照，能更直观地理解 Agent 的工作方式。

上下文压缩函数

这是上下文压缩的核心实现：将旧历史折叠为摘要，保留当前工作现场。

看 GitHub 行号 
 119 # ==================== 上下文压缩（核心新增） ====================
 120 #
 121 # 核心是 compact_messages()；find_recent_start() 用来避免切断 tool 调用组。
 122 #
 123 # 原理：
 124 #   messages = [system, user, assistant, tool, assistant, tool, ...]
 125 #                 ↓ 压缩后
 126 #   messages = [system, 摘要(包含之前所有要点), 最近 N 条消息]
 127
 128 COMPACT_THRESHOLD = 8  # 演示用低阈值：几轮工具调用后就能看到压缩
 129 KEEP_RECENT = 4  # 至少保留最近几条消息；遇到 tool 调用组会向前扩展
 130
 131
 132 def message_role(message):
 133     if isinstance(message, dict):
 134         return message.get("role", "unknown")
 135     return getattr(message, "role", "unknown")
 136
 137
 138 def find_recent_start(messages):
 139     start = max(1, len(messages) - KEEP_RECENT)
 140     # 不要从 tool 消息中间切开；tool 必须紧跟触发它的 assistant tool_call。
 141     while start > 1 and message_role(messages[start]) == "tool":
 142         start -= 1
 143     return start
 144
 145
 146 def compact_messages(messages):
 147     """
 148     当 messages 过长时，把旧消息压缩成一段摘要。
 149
 150     压缩前: [system, msg1, msg2, msg3, msg4, msg5, msg6, msg7, msg8]
 151     压缩后: [system, summary_of(msg1~msg4), ack, msg5, msg6, msg7, msg8]
 152     """
 153     if len(messages) <= COMPACT_THRESHOLD:
 154         return messages  # 没超阈值，不压缩
 155
 156     print(
 157         f"\n[Compact] messages 数量 ({len(messages)}) 超过阈值 ({COMPACT_THRESHOLD})，开始压缩..."
 158     )
 159
 160     system_msg = messages[0]  # system prompt 永远保留
 161     recent_start = find_recent_start(messages)
 162     old_messages = messages[1:recent_start]  # 需要被压缩的旧消息
 163     recent_messages = messages[recent_start:]  # 最近的消息保留原样
 164
 165     # 把旧消息拼成文本，交给 LLM 做摘要
 166     old_text = ""
 167     for msg in old_messages:
 168         role = message_role(msg)
 169         content = (
 170             msg.get("content", "")
 171             if isinstance(msg, dict)
 172             else getattr(msg, "content", "")
 173         )
 174         if content:
 175             old_text += f"[{role}]: {content}\n"
 176
 177     # 调用 LLM 生成摘要
 178     summary_response = client.chat.completions.create(
 179         model=MODEL,
 180         messages=[
 181             {
 182                 "role": "system",
 183                 "content": "Summarize the following conversation history into a concise summary. Keep all important facts, file paths, command results, and decisions. Be concise but don't lose critical details.",
 184             },
 185             {"role": "user", "content": old_text},
 186         ],
 187     )
 188     summary = summary_response.choices[0].message.content
 189
 190     print(
 191         f"[Compact] {len(old_messages)} 条旧消息 → 1 条摘要 (保留最近 {len(recent_messages)} 条)"
 192     )
 193     print(f"[Compact] 压缩后 messages: {1 + 2 + len(recent_messages)} 条\n")
 194
 195     # 重新组装：system + 摘要 + 最近消息
 196     return [
 197         system_msg,
 198         {"role": "user", "content": f"[Previous conversation summary]: {summary}"},
 199         {
 200             "role": "assistant",
 201             "content": "Understood. I have the context from our previous conversation. Let me continue.",
 202         },
 203         *recent_messages,
 204     ]

再看它怎么跑

压缩并非加分项，而是 Agent 完成长任务的关键能力。

演示命令

python agent/06-compact/agent-compact.py "请按步骤执行，不要合并成一个 shell 命令：1. 用 execute_bash 列出 agent 目录下的 Python 文件；2. 分别用 read_file 读取 agent/01-essence/agent-essence.py、agent/02-memory/agent-memory.py、agent/06-compact/agent-compact.py；3. 把三个文件的大致行数写入 compact-demo-report.txt。"
  • 默认演示阈值已调低到 COMPACT_THRESHOLD = 8，分步工具调用后可清晰看到 [Compact] 日志。
  • 压缩会保留 system prompt 与最近几条消息；如果遇到 tool 调用组，会向前扩展边界，避免切断工具响应。
  • 压缩本身也消耗 token，因此是一种工程折中，并非零成本能力。

Try It

自己试一轮

默认低阈值配合分步工具调用，是最直观的压缩演示方式。

  1. 先直接运行演示命令，观察默认低阈值如何触发压缩。
  2. 将 KEEP_RECENT 从 4 改为 2，对比当前任务衔接的变化。
  3. 观察压缩前后 messages 的结构变化，而不只查看日志。

Takeaway

这一讲会看懂什么

  • 理解为何更大的 context window 并非根本解决方案。
  • 能解释 COMPACT_THRESHOLD 与 KEEP_RECENT 的取舍。
  • 理解“保留要点、舍弃细节”是 Agent 的必要能力。

Key Points

三个核心要点

  • 压缩并非为了形式优雅，而是为了避免在真实任务中因上下文溢出而中断。
  • 需要优先保留的是 system prompt 与当前工作现场。
  • 将旧消息摘要化，本质是借助模型整理自身的历史。

Pitfalls

容易误解

  • 摘要过度会丢失关键路径与文件名。
  • recent window 过短会让 Agent 立即丢失工作上下文。
  • 只截断而不压缩，仍无法支撑复杂长任务。
